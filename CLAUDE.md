# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Talaria is a native SwiftUI front-end for [Hermes Agent](https://github.com/NousResearch/hermes-agent). The app renders Hermes natively (instead of embedding its TUI) and shares all protocol/transport logic through the `HermesKit` Swift package so the core stays portable across the macOS and iOS app targets. Hermes is the source of truth for ACP behavior, dashboard HTTP routes, CLI surfaces, and version/capability gates — when behavior is ambiguous, check Hermes, not this app.

## Build & Test

Project files are generated by **XcodeGen** from `project.yml` — that file (not `Talaria.xcodeproj`) is the source of truth for targets, settings, and source membership. `Talaria.xcodeproj` is **git-ignored** (it was a constant merge-conflict point), so run `xcodegen generate` after cloning and after adding/removing/moving source files. The top-level `Talaria.xcworkspace` is hand-maintained and stays tracked; CI and `scripts/release.sh` run `xcodegen generate` before building.

```sh
# HermesKit package tests — fastest loop, covers most logic. Prefer this.
swift test --package-path HermesKit

# A single package test (or a whole suite)
swift test --package-path HermesKit --filter HermesKitTests.SomeTestName

# macOS app build, no signing
xcodebuild build -project Talaria.xcodeproj -scheme Talaria \
  -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO

# macOS app unit tests (TalariaTests target — app-level harness tests)
xcodebuild test -project Talaria.xcodeproj -scheme Talaria -destination 'platform=macOS'

# Shared package built for iOS Simulator (catches iOS-only seam breakage)
xcodebuild build -scheme HermesKit \
  -destination 'generic/platform=iOS Simulator' -workspace Talaria.xcworkspace
```

Release builds (signing, notarisation, stapling, DMG, Sparkle appcast) go through `scripts/release.sh`; see `docs/release.md`. `docs/test-plan.md` is the manual QA script run before a release candidate.

## Architecture

### Read / write model (the central design constraint)

There are two distinct data paths, and code must use the right one:

1. **Live chat** speaks **JSON-RPC 2.0 over a WebSocket** to the dashboard's `/api/ws` gateway — the same path Hermes Desktop uses (`GatewayChatClient` over a `GatewayWebSocket`; `URLSessionGatewayWebSocket` on macOS, `NIOSSHGatewayWebSocket` over the NIO-SSH `direct-tcpip` tunnel on iOS). It rides the **same `hermes dashboard`** the non-chat surfaces use (no separate `hermes acp` subprocess). The gateway events map onto the `HermesNotification`/`SessionUpdate` model in `ChatModels`. See `docs/gateway-chat.md`. (Historical note: chat used to run over an ACP subprocess + a bespoke byte-`Transport` stack; that was removed once the WebSocket path reached parity.)
2. **Everything else** (sessions, skills, cron, logs, updates, profile config) is backed by the **Hermes dashboard HTTP API** — Talaria spawns `hermes dashboard` on an ephemeral loopback port and talks to `/api/*`. Talaria **never reads or writes Hermes SQLite files directly.** Live chat (#1) rides this same dashboard.
3. A few operations have **no dashboard route yet** and fall back to spawning the CLI: `hermes sessions rename`, `hermes tools enable/disable/list`, `hermes doctor`, and the Skills Hub mutations `hermes skills install/update/uninstall` (plus `skills list`/`check` reads). Skills Hub **search** is *not* a CLI fallback — it reads the public Nous index over HTTP (`SkillsHubCatalog`), so it works on iOS too.
4. One deliberate, documented exception to "never read Hermes files directly": the Environment screen **reads the Hermes `.env` directly** (local filesystem, or remote SSH `exec cat` via `RemoteSnapshotTransfer`; path from `hermes config env-path`) purely to *enumerate* user-named custom vars that `GET /api/env` doesn't list. All env **mutations** still go through the dashboard `PUT`/`DELETE /api/env`. See `docs/security.md`.
5. **Terminal (TUI) sessions** (macOS only) are a separate path that bypasses *both* the gateway chat path and the dashboard: a chat can be opened as the real `hermes chat --tui` (resume = `-r <id>`) spawned directly in a PTY and rendered by SwiftTerm. Local = `env hermes … chat --tui`; remote = system `ssh -tt` **always** (even with the NIO opt-in — NIO can't drive a local PTY). Per-launch only (no global default); one mode per session id at a time. Command builder is pure in `HermesKit` (`Transport/TUILaunchSpec.swift`); the view + process registry live in the macOS seam (`Chat/macOS/HermesTerminalView.swift`). SwiftTerm is a macOS-app-only dependency (not iOS, not HermesKit). See `docs/architecture.md`.

`DashboardSupervisor` owns one `hermes dashboard` child: it polls `/api/status`, scrapes the session token from the dashboard SPA, reference-counts consumers, and kills the child when the last one releases. `DashboardCoordinator` caches one supervisor per `(ServerProfile, Hermes profile)` pair — every window shares the default-profile dashboard, while editing a *named* profile in the config editor spins up an isolated, profile-scoped dashboard (`hermes -p <name> dashboard`).

### Window model

Each app window is scoped to one `ServerProfile` and owns its chat session clients (over the dashboard gateway), dashboard client reference, CLI fallback runner, version cache, and capability table. Features gate on the connected Hermes version (e.g. dashboard cron requires `>= 0.14.0`); below the gate, surfaces show a "dashboard required" banner rather than breaking.

### HermesKit modules (`HermesKit/Sources/HermesKit/`)

- `ACP` — Codable JSON-RPC + the session/content/tool/permission schema types (`HermesNotification`, `SessionUpdate`, `ToolCall`, …) the chat UI consumes. (Named for the original ACP protocol; the types are the shared chat-event model the gateway client maps onto.)
- `Transport` — shared pure-Swift NIO-SSH primitives (connection factory, credential providers, host-key trust/TOFU, `OneShotProcess`) used by the dashboard connection, the admin/command runners, and the snapshot transfer. (No longer a chat transport — see `Dashboard` for the chat socket.)
- `Client` — `SessionManager` (per-session lifecycle + notification fan-out/replay) and the `ChatBackend` seam; live chat uses `GatewayChatClient`.
- `Dashboard` — HTTP client, token/session handling, `DashboardSupervisor`, spawn specs, update polling, and the chat WebSocket (`GatewayWebSocket` + `URLSessionGatewayWebSocket` / `NIOSSHGatewayWebSocket`).
- `Hermes` — CLI fallbacks, version parsing, doctor/tool output parsers, capability gates.
- `Profiles` — local and SSH `ServerProfile` models; `HermesConfigDocument` parses each profile's `config.yaml` via Yams **preserving mapping order**.
- `Support` — shared utilities.

### Remote access

macOS reaches remote dashboards by spawning system `ssh` with a loopback `-L` forward and running `hermes dashboard` remotely. iOS uses the pure-Swift NIO-SSH transport: one connection both execs `hermes dashboard` and tunnels its HTTP over a `direct-tcpip` channel (no local forward). Host-key trust uses TOFU — a first connect to an unknown host raises `SSHTransportError.hostKeyUnknown(fingerprint:)` and the UI must present a confirm sheet before `PinnedHostKeyStore.pin(...)`; mismatches and `@revoked` known_hosts entries are never silently re-pinned. See `docs/security.md` for the full trust model and known v1 gaps (encrypted keys, no ssh-agent/`~/.ssh/config` on the NIO path).

## Platform seam convention (important)

The macOS and iOS app targets compile the **same `Talaria/` source tree** with mirror-image excludes (`project.yml`):

- Any folder named `iOS/` under `Talaria/` is **iOS-target-only** (excluded from the macOS target).
- Any folder named `macOS/` under `Talaria/` is **macOS-target-only** (excluded from the iOS target).

This lets a seam file define the **same symbol** in both `macOS/` and `iOS/` folders without an `#if` — only one variant compiles per target. See paired files like `Talaria/Platform/{macOS,iOS}/PlatformSeam.swift` and `Talaria/System/{macOS,iOS}/LogConsoleToolbar.swift`. When adding platform-specific behavior, prefer this folder seam over inline `#if os(...)`.

## Security constraints

- The app is **not sandboxed** (it must spawn `hermes` and `ssh`); it ships with Hardened Runtime + Developer-ID signing + notarisation. `Talaria/Talaria.entitlements` is canonical — adding entitlements needs a blast-radius review.
- Do not store SSH passphrases; delegate auth to system `ssh`, ssh-agent, and `~/.ssh/config` (macOS) or the host-key/identity stores (iOS).
- Log protocol frames only at debug level; never log credential material.

## UI conventions

### Toolbar buttons

Every toolbar button MUST have a tooltip via the `.help("...")` modifier. This is a hard requirement — no toolbar button ships without one.

- `.help()` renders a hover tooltip on macOS and supplies an accessibility hint on iOS.
- Icon-only buttons must ALSO keep their `.accessibilityLabel("...")` — `.help()` is the description, the accessibility label is the name.
- Place `.help()` after the button's label/`.disabled()` modifiers:

  ```swift
  Button { editor.save() } label: {
      Label("Save", systemImage: "checkmark.circle")
  }
  .disabled(!editor.canSave)
  .help("Save the configuration")
  ```

- Applies to every toolbar surface (`.toolbar { ToolbarItem/ToolbarItemGroup }` on both platforms) and toolbar-like control rows (e.g. the add/duplicate/delete strip under the desktop profile list).
- Write the help text as a short action phrase describing what the button does ("Reload the configuration"), not a restatement of the label.
